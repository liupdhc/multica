# Multica 日常运维

站点：`https://multica.oppobit.com`（容器自部署）。

本文依据仓库内官方说明整理，便于现场排障：

- `apps/docs/content/docs/auth-setup.zh.mdx`
- `apps/docs/content/docs/troubleshooting.zh.mdx`
- `docker-compose.selfhost.yml`
- `server/internal/service/email.go`

---

## 1. 登录方式说明

网页端默认是 **Email + 6 位验证码**。可选再配 Google OAuth。

发信优先级（backend 启动时日志会标明当前走哪条）：

| 启动日志 | 含义 |
|---|---|
| `EmailService: SMTP relay <host>:<port> …` | 走 SMTP（`SMTP_HOST` 非空时优先于 Resend） |
| `EmailService: Resend API …` | 走 Resend |
| `EmailService: DEV mode — codes printed to stdout …` | **未配任何邮件后端** |

未配 `RESEND_API_KEY` 且未配 `SMTP_HOST` 时：server **不报错**，验证码**只打印到 backend 容器 stdout**，邮箱收不到。

---

## 2. 未配邮箱时如何登录（紧急/内测）

1. 打开站点，点「开始使用」，输入邮箱并发送验证码。
2. 在部署机查看 backend 日志，抄出 6 位码：

```bash
# 推荐：跟随日志并过滤
docker compose -f docker-compose.selfhost.yml logs -f backend | grep '\[DEV\] Verification code'

# 或按容器名
docker logs -f <backend容器名> 2>&1 | grep '\[DEV\] Verification code'
```

期望输出示例：

```text
[DEV] Verification code for you@example.com: 123456
```

3. 将日志中的验证码填回网页完成登录。

验证当前邮件模式：

```bash
docker compose -f docker-compose.selfhost.yml logs backend | grep "EmailService:"
docker compose -f docker-compose.selfhost.yml exec backend env | grep -E 'RESEND_|SMTP_'
```

若启动日志里没有 `EmailService:` 行，说明相关环境变量未进入进程，检查部署目录的 `.env` 后重启 backend。

## 3. 正式配置发信（推荐）

生产应配置其一，让用户真正收到邮件。

### 方案 A：Resend（公网推荐）

在 `.env` 中设置后重启 backend：

```bash
RESEND_API_KEY=re_xxxxxxxxxxxxxxxx
RESEND_FROM_EMAIL=noreply@yourdomain.com   # 必须是 Resend 已验证域名
```

### 方案 B：SMTP（内网/自有中继）

```bash
SMTP_HOST=smtp.example.com
SMTP_PORT=587                    # 25 匿名 / 587 STARTTLS / 465 隐式 TLS
SMTP_USERNAME=...                # 匿名内部 relay 可留空
SMTP_PASSWORD=...
SMTP_TLS=                        # 465 可省略；非标准 SMTPS 端口设 implicit
SMTP_TLS_INSECURE=false
SMTP_EHLO_NAME=mail.yourdomain.com   # 公网严格 relay（如 Gmail smtp-relay）建议设 FQDN
RESEND_FROM_EMAIL=noreply@yourdomain.com   # 同时作为 SMTP From
```

`SMTP_HOST` 非空时一律走 SMTP，即便也配了 `RESEND_API_KEY`。

改完后：

```bash
docker compose -f docker-compose.selfhost.yml up -d backend
docker compose -f docker-compose.selfhost.yml logs backend | grep "EmailService:"
```

应看到 `SMTP relay` 或 `Resend API`，而不再是 `DEV mode`。

## 4. 固定测试验证码（仅内网/开发）

```bash
APP_ENV=development
MULTICA_DEV_VERIFICATION_CODE=888888
```

注意（硬约束）：

- `docker-compose.selfhost.yml` 默认 `APP_ENV=production`。
- **`APP_ENV=production` 时固定码会被忽略**。
- **不要在公网可访问实例启用固定验证码。**

公网站点 `multica.oppobit.com`：保持 `MULTICA_DEV_VERIFICATION_CODE` 为空；未配邮件时从日志抄码，或尽快配好 Resend/SMTP。

## 5. 邮件收不到时排查清单

1. 看启动日志当前 provider（见第 1 节表格）。
2. 搜是否落回 DEV：

```bash
docker compose -f docker-compose.selfhost.yml logs backend | grep '\[DEV\] Verification code'
```

有这条 = 没配成功邮件后端，码只在日志里。

3. Resend 路径：日志搜 `failed to send verification code`；核对 API key、域名是否已在 Resend 验证。
4. SMTP 路径：日志搜 `failed to send verification email` / `failed to send invitation email`，按阶段修（连通、证书、认证、relay 权限等）。
5. 容器内测连通（SMTP）：

```bash
docker compose -f docker-compose.selfhost.yml exec backend sh -c 'nc -vz $SMTP_HOST $SMTP_PORT'
```

---

## 6. 注册白名单（首次登录被拒时）

相关变量（`docker-compose.selfhost.yml`）：

- `ALLOW_SIGNUP`（默认 `true`）
- `ALLOWED_EMAILS`
- `ALLOWED_EMAIL_DOMAINS`

任一邮箱/域名白名单非空时，不匹配的邮箱**无法首次注册**；已存在用户仍可再次登录。

## 7. 常用命令速查

```bash
# 看 backend 是否在跑、最近日志
docker compose -f docker-compose.selfhost.yml ps
docker compose -f docker-compose.selfhost.yml logs --tail=200 backend

# 重启 backend（改 .env 后）
docker compose -f docker-compose.selfhost.yml up -d backend

# 确认邮件相关环境变量是否进容器（会打印密码，仅在可信终端执行）
docker compose -f docker-compose.selfhost.yml exec backend env | grep -E 'APP_ENV|RESEND_|SMTP_|MULTICA_DEV'
```
